11:T13be,<p>本以为"在 GitLab 编写代码、自动构建，然后一键部署到 GitHub Pages 展示"是一个半小时就能搞定的简单任务。</p>
<p>然而，在打通 <strong>"GitLab CI（加工厂） ➡️ GitHub Pages（展示柜）"</strong> 链路的过程中，我结结实实地踩了四个大坑。以下是我的血泪排障记录，希望帮大家少走弯路。</p>
<p>---</p>
<h3>坑 1：GitLab Runner 权限不足（Docker-in-Docker 报错）</h3>
<p>* <strong>现象</strong>：流水线刚启动，Docker 守护进程就报错崩溃：</p>
<p>  ```text</p>
<p>  mount: permission denied (are you root?)</p>
<p>  Cannot connect to the Docker daemon at tcp://docker:2375.</p>
<p>  ```</p>
<p>* <strong>原因</strong>：自建的 GitLab Runner 默认没有开启特权模式，导致嵌套的 Docker 容器无法挂载系统安全目录。</p>
<p>* <strong>解法</strong>：修改宿主机上的 <code>/etc/gitlab-runner/config.toml</code>，在 <code>[runners.docker]</code> 下面将特权模式开启：</p>
<p>  ```toml</p>
<p>  privileged = true</p>
<p>  ```</p>
<p>---</p>
<h3>坑 2：样式丢失，网页白屏（Jekyll 屏蔽 <code>_next</code>）</h3>
<p>* <strong>现象</strong>：网页部署成功，但打开全是白屏，查看控制台，发现 <code>_next/static</code> 下的所有 CSS 和 JS 文件全部 404。</p>
<p>* <strong>原因</strong>：GitHub Pages 默认用 Jekyll 引擎解析静态网页，而 Jekyll 会<strong>自动忽略任何下划线 <code>_</code> 开头的文件夹</strong>（包括 Next.js 核心的 <code>_next</code>）。</p>
<p>* <strong>解法</strong>：必须在部署根目录下生成一个空文件 <strong><code>.nojekyll</code></strong>，强行禁用 Jekyll 引擎。</p>
<p>---</p>
<h3>坑 3：通配符 <code>*</code> 漏掉了点（隐藏）文件</h3>
<p>* <strong>现象</strong>：明明在脚本里写了 <code>touch gh-pages/.nojekyll</code>，推送上去后依然 404，发现该文件根本没被提交。</p>
<p><em> <strong>原因</strong>：传统的 <code>cp -r out/</em> gh-pages/</code> 拷贝命令，其中的 <code>*</code> 默认<strong>无法匹配到以 <code>.</code> 开头的隐藏文件</strong>（如 <code>.nojekyll</code>）。</p>
<p>* <strong>解法</strong>：改用末尾带点号的拷贝命令：</p>
<p>  ```bash</p>
<p>  cp -r out/. gh-pages/</p>
<p>  ```</p>
<p>---</p>
<h3>坑 4：GitLab Pages 本地静态目录"套娃"</h3>
<p>* <strong>现象</strong>：部署在 GitLab Pages 备份站点的网页也 404 了。</p>
<p>* <strong>原因</strong>：在脚本里直接运行了 <code>cp -r out/ public/</code>。但由于 Next.js 源码中原本就有一个 <code>public</code> 文件夹，导致 <code>cp</code> 命令自作聪明地把整个 <code>out</code> 拷贝到了 <code>public</code> 的内部（变成了 <code>public/out/index.html</code>）。</p>
<p>* <strong>解法</strong>：拷贝前，先无情地强制清理掉原有目录：</p>
<p>  ```bash</p>
<p>  rm -rf public && cp -r out/ public/</p>
<p>  ```</p>
<p>---</p>
<h3>💡 终极版 <code>.gitlab-ci.yml</code> 部署脚本</h3>
<p>在搞定上述问题，并保证 <code>GITHUB_TOKEN</code> 变量名对齐后，我最终磨砺出了这一套能<strong>完美兼顾代码体检（Lint）、构建、并同时双发部署到 GitLab & GitHub</strong> 的极简配置：</p>
<pre><code>image: node:20

stages:
  - lint
  - build
  - deploy

cache:
  key:
    files:
      - package-lock.json
    prefix: $CI_COMMIT_REF_SLUG
  paths:
    - node_modules/
    - .next/cache/

lint:
  stage: lint
  script:
    - npm ci
    - npm run lint
  only:
    - main

build:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - out/
    expire_in: 1 hour
  only:
    - main

pages:
  stage: deploy
  script:
    - rm -rf public
    - cp -r out/ public/
  artifacts:
    paths:
      - public/
  only:
    - main

deploy-github:
  stage: deploy
  script:
    - git clone --depth 1 https://baobanxian:${GITHUB_TOKEN}@github.com/baobanxian/baobanxian.github.io.git gh-pages
    - rm -rf gh-pages/*
    - cp -r out/. gh-pages/
    - touch gh-pages/.nojekyll
    - cd gh-pages
    - git config user.name &quot;baobanxian&quot;
    - git config user.email &quot;2793406287@qq.com&quot;
    - git add .
    - git diff --quiet &amp;&amp; git diff --staged --quiet || git commit -m &quot;Deploy $(date '+%Y-%m-%d %H:%M:%S')&quot;
    - git push origin main
  only:
    - main</code></pre>
